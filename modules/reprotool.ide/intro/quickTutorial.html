Quick 5-minutes tutorial

First install the example projects into your workspace.

  1. Select the File / New / Example... command from the main menu.
  2. Now select the Reprotool example projects wizard and click Next.
  3. Click Finish.
The Reprotool example projects should now be installed in your workspace.

Familiarise yourself with the Citimap example project.

  1. Double click the cityMap.swproj in the Citymap example project directory.
  2. Make sure the outline view is displayed.
  3. Double-click the use-cases in the project editor. A new use-case editor will fire up where you can view the selected use-case.

Introduction to the model-checking in Reprotool

The Citymap example project contains already some annotations. You can think of Reprotool annotations as boolean variables. Use-cases in Reprotool projects consist of use-case steps. You can assign annotaions to individual use-case steps. When the Reprotool performs model checking, it firstly initializes all annotaion variables to boolean value false. It then starts executing use-cases. Executing a use-case means executing its use-case steps. If a step is executed that has an annotation, the value of that annotation variable is changed to true.

In every Reprotool project, there is already a set of predefined annotations that you can use. You can, however, define new annotations. The annotaions in a Reprotool project must adhere to the rules that are defined in the project as a set of temporal logic formulas. The predefined annotations already have some predefined default behaviour. One of the predefined annotations is the create annotation. One of the rules that defines its behaviour in Reprotool is this:

  1. You can not create the same object more than once.
In the next section of this tutorial, you will run the model checking on the Citymap example project and the model checker will generate a counter-example where you can see that there is a possible execution scenario where the zoom object is created more than once. The generated counter-example contains the trace of this execution scenario.

Run the model-check of the Citymap example project.

  1. Right-click the cityMap.swproj file in the project explorer and run the command Convert SW project to SMV from the context menu.
  2. Notice in the project explorer, that a new file cityMap.swproj.nusmv has been generated. View the file. If you are asked whether you want to add the Xtext nature to the project, click yes.
  3. Right-click the cityMap.swproj.nusmv file in the project explorer and run the Run NuSMV verification command from the context menu.
  4. The counter-example trace has been printed on the console output. Also notice in the project explorer, that a new file cityMap.swproj.cexmp has been generated.
  5. View the generated counter-example file cityMap.swproj.cexmp.
  6. Notice in the outline-view of the counter-example trace, that the create_zoom annotation appears two times in the red line that shows the execution path. This is a violation of one of the default rules that bind the create annotation.
